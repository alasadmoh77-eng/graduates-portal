# مواصفات حالات الاستخدام (Use Case Specifications)

يحتوي هذا الملف على تفاصيل ومواصفات حالات الاستخدام الرئيسية لنظام **بوابة خدمات الخريجين - جامعة إقليم سبأ**، موضحاً الفاعلين، الشروط المسبقة، التدفقات الأساسية، والتدفقات البديلة والاستثنائية لكل عملية.

---

## 1. مواصفة حالة الاستخدام: تقديم طلب وثيقة (Submit Document Request)

| البند | التفاصيل |
| :--- | :--- |
| **معرّف حالة الاستخدام** | UC-01 |
| **اسم حالة الاستخدام** | تقديم طلب وثيقة تخرج (سجل أكاديمي أو شهادة تقديرات) |
| **الفاعل الرئيسي** | الخريج (Graduate) |
| **الفاعلون الثانويون** | النظام (System)، المسؤول المالي (Finance Admin) |
| **الوصف** | يقوم الخريج بتقديم طلب للحصول على وثيقة أكاديمية، وتعبئة البيانات المطلوبة، وإرفاق صورة من إثبات دفع الرسوم (إذا كانت الوثيقة مدفوعة). |
| **الشروط المسبقة** | 1. أن يملك الخريج حساباً نشطاً على البوابة.<br>2. أن يكون السجل الأكاديمي للخريج مدخلاً مسبقاً في النظام من قبل القسم المختص. |
| **الشروط اللاحقة** | يتم حفظ الطلب في قاعدة البيانات بحالة `SUBMITTED` وإشعار المسؤول المالي لمراجعة الدفع. |

### التدفق الرئيسي للأحداث (Basic Flow):
1. يقوم الخريج بتسجيل الدخول إلى البوابة.
2. يختار الخريج خيار "طلب وثيقة جديدة".
3. يحدد الخريج نوع الوثيقة المطلوبة (سجل أكاديمي أو شهادة تقديرات) واللغة (عربي/إنجليزي) والغرض من الطلب.
4. يقوم النظام بالتحقق من وجود السجل الأكاديمي للخريج.
5. يعرض النظام الرسوم المطلوبة ورقم الحساب البنكي للإيداع.
6. يقوم الخريج برفع صورة إثبات الدفع (سند الإيداع/التحويل).
7. يضغط الخريج على زر "إرسال الطلب".
8. يقوم النظام بحفظ الطلب برمز تتبع فريد (Tracking Code) وتغيير الحالة إلى `SUBMITTED`.
9. يرسل النظام إشعاراً للمسؤول المالي بوجود إثبات دفع جديد للمراجعة.

### التدفق البديل والاستثناءات (Alternative Flows):
* **خطأ الخطوة 4 (السجل الأكاديمي غير متوفر):**
  * يعرض النظام رسالة تنبيه: "عذراً، لم يتم إدخال بيانات سجلّك الأكاديمي بعد، يرجى التواصل مع إدارة القبول والتسجيل".
  * يتم إلغاء العملية ولا يُسمح بتقديم الطلب.
* **خطوة 6 (الوثيقة مجانية أو معفاة من الرسوم):**
  * يتجاوز النظام خطوة رفع إثبات الدفع.
  * يتم حفظ الطلب مباشرة وتنتقل الحالة إلى `APPROVED` لبدء المراجعة الأكاديمية دون المرور بالمالية.

## 2. مواصفة حالة الاستخدام: مراجعة إثبات الدفع (Review Payment Proof)

| البند | التفاصيل |
| :--- | :--- |
| **معرّف حالة الاستخدام** | UC-02 |
| **اسم حالة الاستخدام** | مراجعة واعتماد إثبات الدفع |
| **الفاعل الرئيسي** | المسؤول المالي (Finance Admin) |
| **الفاعلون الثانويون** | النظام (System)، الخريج (Graduate) |
| **الوصف** | يقوم المسؤول المالي بمراجعة سندات الدفع المرفقة بالطلبات ومطابقتها مع الحساب البنكي للجامعة ثم اعتمادها أو رفضها. |
| **الشروط المسبقة** | أن يكون الطلب في حالة تقديم جديد `SUBMITTED` وحالة الدفع `pending_review`. |
| **الشروط اللاحقة** | في حال القبول: تتحول حالة الدفع إلى `approved` وحالة الطلب إلى `UNDER_REVIEW`.<br>في حال الرفض: يظل الطلب معلقاً مع إرسال تنبيه للخريج لإعادة الرفع. |

### التدفق الرئيسي للأحداث (Basic Flow):
1. يقوم المسؤول المالي بالدخول إلى لوحة التحكم المالية.
2. يستعرض المسؤول قائمة الطلبات بانتظار التحقق المالي.
3. يفتح تفاصيل الطلب ويعرض صورة إثبات الدفع المرفوعة.
4. بعد التأكد من صحة العملية البنكية، يضغط المسؤول على زر "اعتماد الدفع".
5. يقوم النظام بتحديث سجل الطلب (`payment_status = approved` و `status = UNDER_REVIEW`).
6. يرسل النظام إشعاراً تلقائياً للخريج يفيد باعتماد الدفع وانتقال طلبه للمراجعة الأكاديمية.

### التدفق البديل والاستثناءات (Alternative Flows):
* **خطوة 4 (إثبات الدفع غير صحيح أو غير واضح):**
  * يضغط المسؤول المالي على زر "رفض الدفع".
  * يطلب منه النظام إدخال سبب الرفض (مثلاً: الصورة غير واضحة أو مبلغ السند ناقص).
  * يقوم النظام بتحديث حالة الدفع إلى `rejected` وحفظ سبب الرفض.
  * يرسل النظام إشعاراً للخريج بالرفض مع إرفاق السبب ليتمكن من رفع سند جديد.

## 3. مواصفة حالة الاستخدام: المراجعة الأكاديمية (Academic Review)

| البند | التفاصيل |
| :--- | :--- |
| **معرّف حالة الاستخدام** | UC-03 |
| **اسم حالة الاستخدام** | المراجعة والتدقيق الأكاديمي للبيانات |
| **الفاعل الرئيسي** | المسؤول الأكاديمي (Academic Admin) |
| **الفاعلون الثانويون** | النظام (System)، الموقّعون (Signers) |
| **الوصف** | يقوم المسؤول الأكاديمي بمراجعة ومطابقة درجات ومعدل الخريج المدخل في النظام مع الكشوفات الورقية الرسمية للتأكد من سلامتها قبل إرسالها للتوقيع. |
| **الشروط المسبقة** | أن يكون الطلب معتمداً مالياً وفي حالة `UNDER_REVIEW`. |
| **الشروط اللاحقة** | 1. تحديث حالة الطلب إلى `APPROVED`.<br>2. إنشاء مسودة الوثيقة (Draft Document) وإرسالها للموقّع الأول في السلسلة. |

### التدفق الرئيسي للأحداث (Basic Flow):
1. يدخل المسؤول الأكاديمي إلى لوحة التدقيق الأكاديمي.
2. يختار الطلب المراد تدقيقه.
3. يعرض النظام تفاصيل معدل الطالب وسجله الدراسي والدرجات والمستوى الدراسي.
4. يقوم المسؤول بمطابقة البيانات مع الملفات الورقية المعتمدة من الكلية.
5. بعد التأكد من صحة البيانات، يضغط المسؤول على زر "اعتماد أكاديمي".
6. يقوم النظام بتحديث حالة الطلب إلى `APPROVED` ويقوم باستدعاء خدمة `DocumentIssuanceService` لإنشاء مسودة الوثيقة في جدول `issued_documents`.
7. تنتقل حالة الطلب تلقائياً إلى بانتظار التوقيعات `PENDING_SIGNATURES`.
8. يرسل النظام إشعاراً للموقّع الأول في السلسلة يعلمه بوجود وثيقة تتطلب توقيعه.

### التدفق البديل والاستثناءات (Alternative Flows):
* **خطوة 5 (وجود خطأ في الدرجات أو البيانات الأكاديمية):**
  * يقوم المسؤول الأكاديمي برفض الطلب أكاديمياً أو إعادته للتعديل.
  * يدخل سبب الرفض (مثلاً: تعديل في معدل الفصل الأخير).
  * يقوم النظام بتحديث حالة الطلب إلى `REJECTED` وإرسال إشعار للخريج.

## 4. مواصفة حالة الاستخدام: التوقيع الإلكتروني التسلسلي (Sequential Signing)

| البند | التفاصيل |
| :--- | :--- |
| **معرّف حالة الاستخدام** | UC-04 |
| **اسم حالة الاستخدام** | التوقيع الإلكتروني التسلسلي على الوثائق |
| **الفاعل الرئيسي** | الموقّعون (Signers - عميد، مسجل، إلخ حسب نوع الوثيقة) |
| **الفاعلون الثانويون** | النظام (System)، مدير النظام (Super Admin) |
| **الوصف** | يمر المستند المراد إصداره على سلسلة من المسؤولين للتوقيع عليه إلكترونياً بالتسلسل. لا يستطيع أي مسؤول التوقيع إلا عند وصول الدور إليه. |
| **الشروط المسبقة** | 1. أن تكون حالة الطلب `PENDING_SIGNATURES`.<br>2. أن يكون المستخدم الحالي هو الموقّع المطلوب في الترتيب التسلسلي الحالي للوثيقة. |
| **الشروط اللاحقة** | 1. تسجيل التوقيع وتحديث الموقّع التالي.<br>2. عند اكتمال كافة تواقيع السلسلة، تتحول حالة المستند إلى جاهزة للإصدار `READY`. |

### التدفق الرئيسي للأحداث (Basic Flow):
1. يتلقى الموقّع إشعاراً بوجود طلب توقيع معلق.
2. يدخل الموقّع إلى قائمة "التواقيع المعلقة".
3. يستعرض الموقّع الوثيقة وبيانات الخريج والدرجات.
4. يضغط الموقّع على زر "توقيع المستند".
5. يقوم النظام بإدراج صورة التوقيع الإلكتروني للمسؤول في جدول `document_signatures` مع تسجيل الوقت وعنوان الـ IP.
6. يقوم النظام باستدعاء `DocumentSigningService::getCurrentSigner()` لتحديد الموقّع التالي.
7. يرسل النظام إشعاراً للموقّع التالي في السلسلة.

### التدفق البديل والاستثناءات (Alternative Flows):
* **خطوة 6 (الموقّع الحالي هو الأخير في السلسلة):**
  * يقوم النظام بتحديث حقل `all_signed_at` في جدول `issued_documents`.
  * يتم تغيير حالة الطلب تلقائياً إلى جاهز للإصدار `READY` (أو `ISSUED` مباشرة حسب تهيئة النظام الفعلي).
  * يتم إرسال إشعار لمدير النظام لاعتماد الطباعة والإصدار النهائي.
